## Project

Django 5 app serving a single survey ("questionario") for local merchants about traffic/parking changes in the town center. UI text, model labels, and comments are in Italian — keep new user-facing strings in Italian.

## Commands

Dependencies are managed with Poetry (`package-mode = false`); a local `.venv` exists.

```bash
poetry install                          # install deps
python manage.py runserver              # dev server (needs a Postgres reachable; see env vars below)
python manage.py migrate
python manage.py makemigrations questionario

# Full local stack (Postgres 18 + web on :8000)
docker compose -f stacks/stack-local.yml up --build
```

There is no test suite and no CI lint step. Ruff is configured in `pyproject.toml` (line-length 120, py313): `ruff check .`

Database config: `DATABASE_URL` wins, otherwise built from `POSTGRES_USER/PASSWORD/HOST/DB` (all default to `sonda`, host `localhost`). `.env` is read by the Docker stack, not by `manage.py` directly. `db.sqlite3` in the repo root is an empty leftover — the app uses Postgres only.

## Architecture

Three Django apps under one project (`sonda/` is the project package):

- **`questionario/`** — the survey. Single model `Risposta` holds all 10 answers as `CharField`s with `TextChoices` enums defined inline on the model (Q1–Q10 naming convention: field `q<N>_...` matches an enum class). The dashboard view builds frequency tables generically from these enums, so adding/changing a question means touching: the model enum + field, a migration, `RispostaForm` in `forms.py`, and the `freq(...)` call list in `views.dashboard`.
- **`accounts/`** — allauth adapters only. Signup is completely closed; Google OAuth login succeeds only for users that already exist with `is_staff=True` (`SocialAccountAdapter.pre_social_login` connects by email or raises `SignupClosedException`). `/admin/login/` redirects straight to Google OAuth.
- **`sonda/`** — settings, root urls, wsgi.

### Survey access flows (two entry paths)

1. **Public** (`/`): `email_gate` view stores the email in the session, then `questionario` view saves a new `Risposta` and clears the session key.
2. **Invite link** (`/q/<token>/`): the token is a `django.core.signing` signed email (salt `"questionario-invito"`, 30-day expiry, generated by `views.genera_token`). This path is idempotent per email — it loads and updates the existing `Risposta` for that email if one exists. Invite links are generated in the admin (`RispostaAdmin.link_invito` readonly field and a bulk action).

`views.py` owns both token helpers; `admin.py` imports `genera_token` from it.

### HTMX

Views are dual-mode: full-page render normally, partial render when `request.htmx` (via django-htmx middleware). Partials live in `questionario/templates/questionario/partials/`. When changing a view's response, handle both branches.

### Deployment

- `docker/Dockerfile` runs `collectstatic` at build time (WhiteNoise serves static files) and starts via `boot.sh`, which always runs `migrate` then either `runserver` (`DEBUG=1`) or gunicorn.
- `stacks/stack-prod.yml` pulls `ghcr.io/marcotn/sonda:main` and expects `POSTGRES_PASSWORD`, `SECRET_KEY`, `DOMAIN`, `GOOGLE_CLIENT_ID/SECRET` from the environment; it attaches to an external `proxy` network (reverse proxy terminates TLS — settings trust `X-Forwarded-Proto`).
